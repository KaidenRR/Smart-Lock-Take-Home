# Smart Lock Processor — Architecture

## Pipeline

```
┌─────────────┐    ┌──────────────┐    ┌──────────────────┐
│  Smart Lock │ →  │ AWS IoT Core │ →  │  Lambda Handler  │
│  (firmware) │    │  Rules       │    │  (this repo)     │
└─────────────┘    └──────────────┘    └────────┬─────────┘
                                                │
            ┌───────────────────┬───────────────┼───────────────┐
            ▼                   ▼               ▼               ▼
    ┌───────────────┐   ┌──────────────┐  ┌──────────┐  ┌─────────────────┐
    │ Dedup table   │   │  Unit table  │  │ Device   │  │  IoT Publish    │
    │ (DynamoDB)    │   │  (DynamoDB)  │  │ Shadow   │  │  → Dispatcher   │
    └───────────────┘   └──────────────┘  └──────────┘  └─────────────────┘
```

## MQTT Topic Format

```
v/{tenant_id}/{unit_id}/lock/{event_type}
```

Five segments separated by `/`. Examples:

```
v/acme/7720fc98-1257-4a2e-9abb-55dc3dd5a54b/lock/state_change
v/acme/7720fc98-1257-4a2e-9abb-55dc3dd5a54b/lock/jammed
v/acme/7720fc98-1257-4a2e-9abb-55dc3dd5a54b/lock/forced_open
```

## Event Payload

The lock firmware emits MQTT payloads of the form:

```json
{
  "topic": "v/acme/7720fc98-.../lock/jammed",
  "event_id": "evt_01HXYZ7K8M2N3P4Q",
  "state": "jammed",
  "sensor_id": "front_door_lock",
  "device_name": "Front Door Lock",
  "entity_id": "lock.front_door",
  "battery_pct": 42,
  "timestamp": "2026-04-16T18:22:11.000Z"
}
```

`event_id` is generated by the lock's secure element and is **unique per
device-emitted event**. It is preserved across MQTT QoS 1 retransmissions —
i.e., a real retransmission of "the same event" carries the same `event_id`,
while two separate jam-then-cleared-then-jam-again events carry distinct
`event_id`s.

## Data Stores

- **Unit table** — see `infra/terraform/unit_table.tf`. Stores per-unit
  metadata. Schema documented in [`unit-schema.md`](./unit-schema.md).
- **Dedup table** — see `infra/terraform/dedup_table.tf`. Stores recent
  message hashes; TTL-managed.
- **Device shadow** — AWS IoT named shadow per `sensor_id`. Reflects the
  last known state of each lock.

## Concurrency Model

The Lambda is configured with reserved concurrency = 50. Multiple events
for the same unit can be in-flight simultaneously (lock + auto-lock fires
within milliseconds; battery-low report races with state change).

The IoT shadow service performs a partial merge on `reported` natively —
fields not included in your `UpdateThingShadow` payload are preserved. See
[AWS IoT Device Shadow service docs](https://docs.aws.amazon.com/iot/latest/developerguide/device-shadow-data-flow.html)
for merge semantics.

## Downstream

Notification publishes go to `v/{unit_id}/message/default/send` and are
consumed by the `IoT_Email_SMS_Dispatcher` Lambda. Contract for that
publish is in [`dispatcher-contract.md`](./dispatcher-contract.md).
